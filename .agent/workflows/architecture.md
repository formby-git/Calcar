# Calcar Architecture & SSR Flow

> **Instructions for Agents:**
> Read this file to understand the routing, server-side rendering, and data passing architecture of the Calcar web app before attempting to modify core logic or components.

## 1. Web Application Flow (Astro SSR)

Calcar operates on a Server-Side Rendered (SSR) model, meaning pages are generated dynamically on request using Cloudflare Workers.

### Step 1: The Landing Page (`src/pages/index.astro`)
- The user lands on the homepage.
- The `RegistrationForm.astro` component accepts a vehicle registration plate.
- Upon submission, a client-side script intercepts the form, formats the plate, and performs a simple `window.location.href = '/analyse/gb/' + formattedPlate`.

### Step 2: The Analysis Route (`src/pages/analyse/[country]/[registration].astro`)
This is the core dynamic route.
- **Server-Side Execution**:
  1. Retrieves the `[registration]` from `Astro.params`.
  2. Fetches the DVLA API Key from `import { env } from "cloudflare:workers"`.
  3. Awaits `getCarByRegistration(registration, apiKey)`.
- **Error Handling**:
  - If no car is returned (e.g. 404 from DVLA), the server renders a user-friendly "Vehicle Not Found" UI.
- **Rendering**:
  - If successful, the server renders the HTML for `VehicleCard` and `TCOCalculator`.
  - **Crucial Pattern**: These components are rendered *without* props. Instead, a `<script define:vars={{ car }}>` block at the bottom of the analysis page dispatches a global `car-loaded` `CustomEvent` to the `window`.

### Step 3: Component Initialization (`src/components/`)
Astro generally avoids shipping heavy frontend frameworks (like React). Instead, Calcar relies on Vanilla JS inside `<script>` tags within Astro components.
- Components like `VehicleCard` and `TCOCalculator` listen for the `car-loaded` event on the `window`.
- Once the event is received, they read the `event.detail` (which contains the `Car` object) and populate their DOM elements (e.g. updating `innerHTML`, changing image `src`, or performing client-side math).

## 2. Cloudflare Pages Deployment (`_worker.js`)

In late 2024, the `@astrojs/cloudflare` v13 adapter transitioned from legacy "Pages Functions" (`_worker.js` generated by Astro) to a "Wrangler Workers" deployment model. 

Because the native Cloudflare Pages CI does not natively know how to deploy the Wrangler `dist/server/wrangler.json` output, we use a custom build step.

### The Build Step (`package.json`)
```json
"bundle-worker": "esbuild dist/server/entry.mjs --bundle --outfile=dist/client/_worker.js --format=esm --platform=browser --external:cloudflare:* --external:node:*",
"build": "astro build && npm run bundle-worker"
```
1. `astro build` compiles the SSR entry point to `dist/server/entry.mjs`.
2. `bundle-worker` takes that entry point and uses `esbuild` to bundle it into a single `_worker.js` file and places it in `dist/client/`.
3. Cloudflare Pages natively looks for `dist/client/_worker.js` and automatically deploys it as the SSR worker.

### Environment Variables
Environment variables (like `DVLA_API_KEY`) are *not* baked in at build time. They are injected at runtime by Cloudflare and accessed via `import { env } from "cloudflare:workers"`. **Never** use `Astro.locals.runtime` as it was removed in adapter v13.
